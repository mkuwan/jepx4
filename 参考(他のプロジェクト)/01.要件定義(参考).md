# 00. 要件定義（JEPX API連携システム）

## 1. 文書情報

### 1.1 文書目的
本書は、JEPX の会員サイト廃止（API連携への移行）に伴い、翌日市場取引システム・時間前市場取引システム・専用接続線接続仕様を前提とした新システムの要件を定義する。

### 1.2 参照文書
- `API仕様書(時間前市場取引システム).md`（ver.1.0.3）
- `API仕様書(翌日市場取引システム).md`（ver.1.1.1）
- `JEPX 専用接続線 接続技術書.md`（ver.2.1）
- `指示書.md`

### 1.3 用語
- DAH: 翌日市場（Day-Ahead）
- ITD: 時間前市場（Intraday）
- ITN: 市場情報通知（配信）
- 約定: 成立した取引
- ステートレス: 処理結果・照会データをローカルに永続保持せず、必要都度 JEPX API から取得して表示・処理する設計方針
- 監査ログ: 操作者・実行時刻・API 入出力・判定理由等を記録するログ（内部 DB ではなくログファイル/ログサービス）

### 1.4 基本方針（アーキテクチャ原則）

本システムは以下の 3 原則を全要件の前提とする。これと矛盾する要件を設けてはならない。

| 原則 | 内容 |
|------|------|
| **ステートレス処理** | バッチ・画面のいずれも、表示・処理に必要なデータは都度 JEPX API から取得する。処理結果をローカルに保持しない。 |
| **DB 不使用** | リレーショナル DB・NoSQL DB・ローカルキャッシュ DB 等、永続データストアは一切使用しない。コード定義・設定値はファイル（YAML/JSON/ENV）で管理する。 |
| **ログ保存** | 操作・API 通信・判定・エラーの証跡を監査ログとして保存する。ログは監査・障害調査のための証跡であり、業務データとは明確に区別する。 |

## 2. 背景・目的

### 2.1 背景
- 現行業務は Web 画面操作と RPA を中心に、入札・削除・照会・約定参照・結果比較を実施している。
- JEPX は GUI ベース運用を終了し API 中心に移行するため、現行 RPA は継続不能となる。
- 現行業務は、証跡保管・再送手順・例外時の統制・通信障害対応が個別運用に依存している。

### 2.2 目的
- CSV/Excel 取込から API 実行、結果照会、監査、レポートまでを一気通貫で自動化する。
- 冪等性・再送制御・監査証跡・障害復旧（BCP）を標準化し、実運用可能な品質を確保する。
- 翌日市場はバッチ自動実行を基本としつつ、時間前市場と同等の手動操作（画面）も可能とする。
- 時間前市場は全処理を画面操作中心で運用可能とする。
- **DB を使用せず、処理はすべてステートレスとすることで、運用・保守コストと機密データの保持リスクを排除する。**

### 2.3 成功指標（KPI）
- 定常日の入札処理成功率: 100%
- 締切前安全完了: 締切 10 分前までに対象入札処理完了
- API 送信重複事故: 0 件
- 監査ログ欠落: 0 件

## 4. To-Be 業務像

### 4.1 全体業務フロー（標準）
1. 計画値ファイル（CSV/Excel）を取り込む
2. 形式・業務ルール・スケジュール妥当性を検証する
3. API リクエスト電文を生成する
4. API 送信・応答受信を行い、結果をログに記録する
5. 入札照会／約定照会／市場結果照会を JEPX API に都度発行し、結果を画面表示する
6. 比較・集計・レポートを画面表示または出力ファイルとして生成する
7. 操作・通信の証跡をログに保存する

> **注意**: ステップ 4 において処理結果をローカルに保存しない。照会・比較・レポートはいずれも JEPX API への都度リクエストにより最新データを取得して実施する。

### 4.2 実行モード
- **バッチ（主に DAH）**
	- 日次スケジュールで自動実行
	- 締切逆算で処理順序・再試行を自動制御
	- バッチ実行ごとの操作・送受信記録をログに出力する
- **画面（主に ITD、かつ DAH 手動補助）**
	- 各画面を開いた時点で JEPX API を呼び出し最新データを表示する
	- 状態監視、再送、削除、照会、比較、レポートを操作可能
	- 画面セッション中の一時データは永続化しない

### 4.3 例外時フロー
- API エラー時: 事由判定（入力不備/スケジュール外/通信障害/会員不正）
- 再送可否判定: JEPX 照会 API を呼び出して既送信の有無を確認し、送信未到達の場合のみ再送する
- 復旧: 再送、再照会（いずれも JEPX API 呼び出し）
- 監査: 全判断・全 API 通信履歴をログに出力する

## 5. システム構成要件

### 5.1 技術要件（指示書準拠）
- アプリケーション: Python（Django / **ASGI**等） または Java（Spring Boot等）
  - ※ Pythonの場合：リアルタイム配信（ITN）をメモリ共有しつつブロッキングを回避するため、WSGIではなくASGI（Uvicorn等）を採用する。
  - ※ Javaの場合：WebSocketやSpring WebFlux（あるいは非同期Servlet）等を用いてITN配信時の非同期・ノンブロッキング処理に対応する。
- 実行形態: バッチ + Web 画面
- インフラ: AWS 環境（VPC 構成）
- 接続経路: バッチサーバー → FW → 中継サーバー → AWS Direct Connect → JEPX
- データストア: DB 不使用。ログはファイル出力する

### 5.2 論理コンポーネント
- 取込・検証モジュール（メモリ内処理）
- API 通信モジュール（Socket/TLS/gzip/電文）
- 業務制御モジュール（締切管理、再試行、冪等性チェック）
- 画面表示モジュール（都度 API 呼出し・描画）
- 監査ログモジュール（操作・API 入出力・判定の記録）
- 出力モジュール（CSV/Excel/PDF の一時生成・ダウンロード）

### 5.3 データの永続化方針
| 種別 | 扱い |
|------|------|
| 入札計画値（CSV/Excel） | 取込時のみメモリ内処理。 |
| API 送受信データ | ログに出力する |
| 照会結果・約定結果 | 都度 JEPX API から取得して表示。 |
| レポート・出力ファイル | 要求時に一時生成しダウンロード提供。 |
| コード定義・設定値 | YAML/JSON/ENV ファイルで管理 |
| ログ | ファイルに出力する |

## 6. 外部インターフェース要件

### 6.1 接続方式（JEPX 専用接続線）
- 通信方式: TCP/IP Socket 通信
- 暗号化: TLS 1.3
- IP 制限: 各サーバ毎に許可 IP は 1 つ
- Socket 上限: 1 通信線・1 サーバ当たり `5 + 1（配信用は1）`
- 一般通信: 3 分無通信でサーバ側切断
- 配信通信: 無通信でもサーバ側切断なし

### 6.2 電文フォーマット
- 構成: `SOH` + ヘッダ + `STX` + ボディ + `ETX`
- ヘッダ文字コード: ASCII
- ヘッダ形式: `項目名=値` をカンマ区切り
- 要求ヘッダ項目:
	- MEMBER（会員ID）
	- API（機能番号）
	- SIZE（ボディ Byte 数）
- 応答ヘッダ項目:
	- STATUS（電文ステータス）
	- SIZE（ボディ Byte 数）
- ボディ: gzip 圧縮された JSON

### 6.3 電文ステータス対応
- `00`: 電文処理正常（機能エラーはボディ status を参照）
- `10`: 電文フォーマット異常
- `11`: 会員ID権限なし
- `19`: サーバシステム異常

### 6.4 共通 API（接続維持）
- SYS1001（ソケット接続時間延長）を実装すること
- 一般通信で 3 分アイドルに達する前に keep-alive 送信可能であること

## 7. 業務要件（BR）

### BR-01 対象業務完全性
翌日市場・時間前市場の入札、削除、照会、約定、清算の一連業務を API で実施できること。

### BR-02 自動化
翌日市場業務はバッチによる自動処理を基本とし、締切までに安全完了すること。

### BR-03 手動運用性
時間前市場業務は画面から一通り実施でき、状況確認・入札・削除・照会が容易であること。

### BR-04 監査性
入力、送信、応答、判定、時刻をログに出力すること。

### BR-05 事業継続性
通信障害・処理失敗時に、JEPX 照会 API を用いた現況確認を起点として、手順化された復旧運用可能であること。

### BR-06 ステートレス運用
すべての画面表示・バッチ処理は JEPX API から都度データを取得して処理すること。

## 8. 機能要件（FR）

### 8.1 取込・検証

#### FR-01 ファイル取込
- CSV/Excel ファイルを取り込めること
- 文字コード、ヘッダ、必須列、型不一致を検出すること
- 取り込んだデータはメモリ内で処理し、サーバに保存しないこと

#### FR-02 入力バリデーション
- 必須・形式・範囲・単位・コード・日程・不整合を検証すること
- JEPX エラー分類（required / format / inconsistency / unit / range / code / member / schedule / size / none）と対応づけて表示すること
- バリデーション処理はメモリ内で完結させること

#### FR-03 業務妥当性
- 入札価格が 10 の倍数であることを検証すること
- 入札量の小数処理（小数第1位有効、2位以下切捨）を統一すること

### 8.2 API 実行（翌日市場）

#### FR-10 DAH API 実装
以下 API を実行可能とする。
- DAH1001 入札
- DAH1002 入札照会
- DAH1003 入札削除
- DAH1004 約定照会
- DAH9001 清算照会

#### FR-11 DAH 入札制御
- 通常入札の投入順序、再実行順序を制御すること
- 入札削除時に対象指定あり/なし（全削除）を扱えること
- 各 API の送受信内容を監査ログに記録すること

### 8.3 API 実行（時間前市場）

#### FR-20 ITD/ITN API 実装
以下 API を実行可能とする。
- ITD1001 入札
- ITD1002 入札削除要求
- ITD1003 入札照会
- ITD1004 約定照会
- ITD9001 清算照会
- ITN1001 市場情報通知受信

#### FR-21 ITD 特有制御
- 入札削除要求（DEL）の扱いを通常入札と区別して管理すること
- 商品照会の取引中断コードに基づき、対象商品の取引可否を画面表示すること

#### FR-22 配信受信
- 市場情報通知（CONTRACT / BID-BOARD）を継続受信し、最新データを画面に反映すること
- 受信データはAPサーバ（PythonのASGIプロセスやJavaのインスタンス等）内のメモリ（インメモリキュー等）に共有保持し、画面表示(SSE/WebSocket)後は永続保存しないこと（外部DB/Redis等は使用しない）
- 0件通知（市場情報なし）を正常ケースとして扱えること

### 8.4 照会・表示

#### FR-30 照会結果表示
- 入札照会・約定照会・市場結果照会は、画面操作のたびに JEPX API を呼び出して最新データを取得し表示すること
- ローカルへのキャッシュや DB 保存を行わないこと
- API 通信の内容（リクエスト・レスポンス）は監査ログに記録すること

#### FR-31 比較・検算
- 取り込んだ計画値と JEPX API から取得した約定値を照合し、数量・価格・金額の差分をファイル出力すること

#### FR-32 清算表示
- 清算照会（DAH9001/ITD9001）を都度 JEPX API に発行し、清算ファイル・PDF（Base64）をファイル出力すること

### 8.5 運用画面

#### FR-40 操作機能
- 手動入札、削除、照会、再送、再集計、レポート出力を画面から実行できること
- 各操作の実行・結果を監査ログに出力すること


### 8.6 実行制御・冪等性

#### FR-50 二重送信防止
- 同一バッチ実行内での重複送信はプロセスメモリ内の送信済みキーセットで防止すること
- 実行前に JEPX 照会 API（DAH1002/ITD1003）を用いて既登録状態を確認し、既に入札が存在する場合は再送しないこと
- 二重送信確認の照会操作をログに出力すること

#### FR-51 再試行
- 通信失敗時の自動再試行（リトライ）はプロセス内で実施すること
- 業務エラー時の手動再送は画面から実施でき、再送前に JEPX 照会 API で現況を確認すること

#### FR-52 タイムアウト対策
- Socket 3分無通信制約を考慮し、長時間処理中は SYS1001 で接続延長可能であること

### 8.7 帳票・出力

#### FR-60 出力
- CSV/Excel/PDF 出力に対応すること
- 出力ファイルはリクエスト時に JEPX API から取得したデータをもとに都度生成しダウンロード後はサーバに残さないこと
- 日次・市場別・成功失敗別の運用レポートを出力できること（データは JEPX API から都度取得）

## 9. 非機能要件（NFR）

### 9.1 性能
- 翌日市場対象のバッチ実行は、9時までに処理完了すること

### 9.2 可用性
- API 通信失敗時に再送または後続照会で整合回復できること
- 障害時は JEPX 照会 API を状態確認の起点とし、処理停止と再開が可能であること

### 9.3 信頼性
- 電文フレーム（SOH/STX/ETX）、ヘッダ SIZE、gzip 展開の整合性検証を行うこと
- 受信 STATUS とボディの status の両方で成否判定すること

### 9.4 セキュリティ
- TLS1.3 で通信すること
  - ※ 【前提条件】稼働OS（Linux）の OpenSSL バージョンが `1.1.1` 以上（または `3.x` 系）であること。
  - ※ 【Python環境】Python `3.8` 以上（OpenSSL 1.1.1以上とリンクされた実行環境であること）が必須。推奨は `3.10` 以上。
  - ※ 【Java環境】Java `11` 以上（非推奨：Java 8）の実行環境が必須。推奨は `17` 以上。
- IP 制限、会員ID管理、認証情報の環境変数管理を徹底すること

### 9.5 ログ
- すべての証跡はログ（ログファイル / CloudWatch Logs / S3 等）により管理する
- API リクエスト/レスポンス（機密情報は必要マスキング後）をログに出力すること
- 実行時刻・操作内容・判定理由・エラー内容をログに出力すること
- ログは長期保管（対象期間は運用設計で規定）可能な構成とすること
- 異常検知・再送結果・操作履歴をログから追跡可能であること

### 9.6 保守性
- API バージョン差分（仕様改定）に追従しやすい構成であること
- コード定義（エリア、時間帯、種別、エラー）は設定ファイル（YAML/JSON）で管理し、更新時のデプロイのみで反映可能とすること

## 10. ログ要件

本システムは DB を使用しないため、証跡・監査・障害調査はすべてログで行う。

### 10.1 ログ種別

| 種別 | 記録内容 | 用途 |
|------|----------|------|
| 操作ログ | 操作日時・操作内容・対象データキー | 監査・証跡 |
| API 通信ログ | API 種別・リクエスト内容（マスキング済）・レスポンス STATUS・ボディ status・処理時間 | 障害調査・監査 |
| エラーログ | エラー分類・詳細メッセージ・スタックトレース | 障害調査 |
| 実行ログ（バッチ） | バッチ名・開始/終了時刻・処理件数・成功/失敗件数 | 運用監視 |

### 10.2 ログ出力先
- ローカルファイルシステムへ一時出力

### 10.3 ログ保管要件
- 保管期間・アーカイブ・削除ルールは運用設計で規定する
- 長期保管に備えて圧縮・アーカイブの仕組みを整えること
- 本番環境・ステージング環境でのログ分離を行うこと

### 10.4 禁止事項
- ログに平文で機密情報（認証情報、個人情報等）を記録しないこと
- ログをデータの代替 DB として使用しないこと（ログはあくまで証跡であり、業務処理の入力には使用しない）

---

## 11. コード定義・設定管理

DB を使用しないため、マスタデータに相当するコード定義・設定は設定ファイルで管理する。

### 11.1 設定ファイル管理対象
- エリアコード（1〜9）
- 時間帯コード（01〜48）
- 入札種別コード（SELL-LIMIT, BUY-LIMIT, SELL-MARKET, BUY-MARKET, SELL-FIT, SELL-APPROVAL, DEL）
- 取引中断コード（0, 1）
- 通知種別コード（CONTRACT, BID-BOARD）
- API エンドポイント・タイムアウト・リトライ設定
- ロール定義・権限マトリクス

### 11.2 管理方針
- 設定ファイルはバージョン管理（Git）で管理すること
- アプリケーション起動時に読み込み、プロセス内でのみ有効とすること
- 設定変更は再デプロイをもって反映すること

## 12. バッチ要件（翌日市場）

### 12.1 実行シナリオ
1. 計画値ファイル取込（メモリ内）
2. 検証（メモリ内）
3. 入札送信（JEPX API）
4. 入札照会（JEPX API で都度取得）
5. 約定照会（JEPX API で都度取得）
6. 清算照会（JEPX API で都度取得）
7. 比較・出力（JEPX API 取得データとメモリ上の計画値で比較）
8. 操作・通信記録をログに出力

### 12.2 制御要件
- スケジュール定義（営業日、締切逆算、再送ウィンドウ）
- ジョブ排他制御（同一市場・同一受渡日、プロセスロックで管理）
- バッチ内部の処理状態はプロセス内メモリで管理し、完了後は保持しない
- 処理失敗時のリカバリは最初から再実行（JEPX 照会 API で現況確認のうえ、未送信分のみ再送）で対応すること

## 13. 画面要件

### 13.1 主要画面
- ダッシュボード（JEPX API 照会結果をリアルタイム表示）
- 計画値ファイル取り込み（アップロード・メモリ内処理・バリデーション結果表示）
- 入札実行/削除画面
- 照会結果画面（JEPX API 都度取得）
- 比較・レポート画面（JEPX API 都度取得データと計画値の比較・出力）

### 13.2 表示原則
- すべての業務データ表示は JEPX API へのリアルタイムリクエストにより取得すること
- 画面上に表示するキャッシュの有効期間はセッション内の単一リクエスト処理のみとし、永続化しないこと
- 画面表示時の JEPX API 通信エラーはユーザに通知し、再取得ボタン等で対処できること

### 13.3 表示項目（最低限）
- 市場区分、API 種別、受渡日、時間帯、入札番号
- 実行時刻、実行者、結果ステータス、エラーコード
- 再送可否、再送回数、最終確認時刻

## 15. エラー・例外要件

### 15.1 エラー分類
- 入力エラー（required / format / unit / range / code）
- 業務整合エラー（inconsistency / size / none）
- 会員・スケジュールエラー（member / schedule）
- 電文・通信エラー（STATUS 10/11/19）

### 15.2 例外処理
- エラー内容を操作者に明示し、再実行可能な単位で復旧できること
- 再実行前に JEPX 照会 API で現況を確認し、重複送信を防ぐこと
- 失敗データを隔離し、全体停止を最小化すること
- すべてのエラー処理経緯を監査ログに記録すること

## 16. 受入要件（抜粋）

### 16.1 機能受入
- 参照 API を対象に全機能が実行できること
- 取込〜入札〜照会〜比較〜出力の一連処理が成立すること
- 画面上のすべての業務データが JEPX API からリアルタイムに取得されていること

### 16.2 品質受入
- 二重送信が発生しないこと（照会 API による事前確認で検証）
- 監査ログから処理追跡が可能であること
- 通信切断・再接続時の回復動作が確認できること

### 16.3 運用受入
- バッチ定常運用手順が成立すること
- 画面から時間前市場業務を完結できること
- ステートレス設計のためプロセス再起動後も業務継続できること（ローカル状態に依存しないこと）
- 監査ログによる証跡確認手順が成立すること

## 18. 要件定義まとめ

本要件定義では、JEPX API 連携に必要な業務・機能・非機能・運用要件を、翌日市場/時間前市場/専用接続線仕様に整合する形で網羅した。

本システムのすべての業務データは JEPX API から都度取得することで最新性を保つ。

証跡については「ログのみで担保」する方針を採る。操作・通信・判定のすべてをログに記録し、ログサービスにより長期保管することで監査要件を満たす。

特に中核要件として定義した事項：
- **ステートレス処理**: 画面・バッチとも JEPX API 都度取得
- **DAH の自動化**（バッチ中心）
- **ITD の運用性**（画面中心）
- **接続仕様準拠**（TLS1.3、Socket制約、電文フォーマット）
- **冪等性・再送**: JEPX 照会 API を活用した重複防止
- **BCP**: ステートレスゆえプロセス再起動で即時復旧

以降、基本設計・詳細設計では本書の要件IDをトレーサブルに引継ぎ、試験計画まで一貫管理する。
