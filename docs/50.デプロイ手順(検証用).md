# 30. デプロイ手順 (検証環境用)

## 概要
本ドキュメントは、Azure仮想マシン(VM)上に「JEPX API Web UI（DBレス・Azure SSO版）」および「MockServer（Docker版）」を構築し、検証環境として稼働させるための具体的な手順を、初心者でも迷わず実行できるようステップバイステップで記載したものです。
データベースは一切使用せず、認証情報はクライアント側のSigned Cookie内で管理される完全ステートレス構成となります。

---

## 🚀 1. 事前準備 (Azure ポータルでの仮想マシンとSSO基盤設定)

### 1.1 Azure 仮想マシンの作成
1. Webブラウザで [Azure Portal](https://portal.azure.com) にアクセスし、サインインします。
2. 既存のリソースグループ（例: `JEPX_RG`）に入るか、検索バーから「Virtual Machines (仮想マシン)」を開き、新規作成します。
   - **仮想マシン名**: `jepx-stage-vm` など分かりやすい名前。
   - **地域**: `Japan East` (東日本) を推奨。
   - **イメージ**: `Ubuntu Server 24.04 LTS` (または 22.04 LTS)
   - **サイズ**: `Standard_B2s` 等 (MockServerのDockerとWebが同居するため、メモリ4GB以上を推奨)
   - **管理者アカウント**:
     - **認証の種類**: `SSH 公開キー`
     - **ユーザー名**: `azureuser`
     - **キーのペア名**: `jepx-stage-key`
   - **受信ポートの規則**:
     - `HTTP (80)`, `HTTPS (443)`, `SSH (22)` の3つにチェックを入れて許可します。
3. **「確認および作成」** を押し、「新しいキーのペアの生成」ポップアップが出たら、必ず **「プライベート キーのダウンロードとリソースの作成」** をクリックしてください。
   - ダウンロードされた鍵ファイル (`jepx-stage-key.pem`) は後ほどサーバーにログインするために必要ですので、大切に保管してください。
4. デプロイ完了後、対象リソースの画面に移動し **パブリック IP アドレス**（例: `20.xxx.xxx.xxx`）をコピーしてメモしておきます。

### 1.2 独自ドメイン (DNS) の設定
Azureポータル上で「DNSゾーン」を開き、先ほどメモしたパブリックIPアドレスをドメインに紐付けます。（例: `mkuwan.jp` ドメインを使用する場合）
1. リソース一覧から、種類が「DNSゾーン」となっているもの（例: `mkuwan.jp`）を開きます。
2. 画面上部の **「＋ レコード セット」** をクリックします。
3. **名前**: `jepx-stage` (これで `jepx-stage.mkuwan.jp` にアクセスできるようになります)
4. **種類**: `A`
5. **IP アドレス**: 先ほどメモしたVMのパブリックIPアドレスを貼り付けます。
6. **「保存 (OK)」** をクリックします。

### 1.3 Azure Entra ID でのアプリ登録 (SSO準備)
1. Azure Portalの検索バーから **「Microsoft Entra ID」** を開き、左メニューの **「アプリの登録」** へ移動します。
2. **「新規登録」** をクリックし、以下を入力します:
   - 名前: `JEPX_Stage_Web`
   - サポートされているアカウントの種類: 「この組織ディレクトリのみに含まれるアカウント」
   - リダイレクト URI (プラットフォーム: Web): `https://jepx-stage.mkuwan.jp/callback/`
     *(※独自ドメインでHTTPS化するため、必ず `https://` から始まるURLを入力してください)*
3. 登録完了後、遷移した概要ページにて以下の2つをメモします:
   - **アプリケーション (クライアント) ID** (`ENTRA_CLIENT_ID`)
   - **ディレクトリ (テナント) ID** (`ENTRA_TENANT_ID`)
4. 左メニュー **「証明書とシークレット」** > **「新しいクライアントシークレット」** を作成し、表示された **「値 (Secret)」** をメモします (`ENTRA_CLIENT_SECRET`)。※一度しか表示されません。
5. 左メニュー **「APIのアクセス許可」** にて `User.Read` が付与されていることを確認し、「(組織名) に管理者の同意を与えます」をクリックします。

## 💻 2. サーバー環境の基本セットアップ (SSH接続)

### 2.1 SSHを利用した仮想マシンへの接続
お手元のWindowsパソコンで「ターミナル」または「Windows PowerShell」を開き、ダウンロードした鍵を使ってサーバーへ接続します。

```powershell
# 1. 鍵ファイルがダウンロードされたフォルダへ移動 (通常はDownloads)
cd ~\Downloads

# 2. SSHで接続 (yes/no を聞かれたら yes と入力してEnter)
ssh -i "jepx-stage-key.pem" azureuser@<メモしたパブリックIPアドレス>
```

### 2.2 必須ソフトウェアのインストール
ログイン成功後 (`azureuser@jepx-stage-vm:~$` 等の表示になったら)、以下のコマンドをすべてコピー＆ペーストして環境を整えます。

```bash
# OSのソフトウェアリストを最新化し、アップグレード (-yで自動Yes)
sudo apt update && sudo apt upgrade -y

# Python実行環境, Nginx, Certbot (HTTPS化ツール) のインストール
sudo apt install -y python3-pip python3-venv nginx certbot python3-certbot-nginx

# Dockerエンジン等のインストール
sudo apt install -y apt-transport-https ca-certificates curl software-properties-common
curl -fsSL https://download.docker.com/linux/ubuntu/gpg | sudo gpg --dearmor -o /usr/share/keyrings/docker-archive-keyring.gpg
echo "deb [arch=$(dpkg --print-architecture) signed-by=/usr/share/keyrings/docker-archive-keyring.gpg] https://download.docker.com/linux/ubuntu $(lsb_release -cs) stable" | sudo tee /etc/apt/sources.list.d/docker.list > /dev/null
sudo apt update
sudo apt install -y docker-ce docker-ce-cli containerd.io

# 現在のローカルユーザー(azureuser等)がsudoなしでDockerを使えるように権限を付与
sudo usermod -aG docker $USER
```
*(※この権限追加設定を反映させるため、ここでターミナル上で **`exit`** と入力していったんログアウトし、**再度 ssh コマンドでログイン** し直してください。その後 `docker ps` が実行できれば成功です)*

## 🐳 3. MockServerの構築 (Docker)

JEPXの各種取引API・Socket通信を模倣するMockServerを構築します。
本環境ではDocker Hub等の外部レジストリは使わず、ローカルのソースコードをVMへ転送し、VM内で直接Dockerイメージをビルドしてコンテナを起動します。

### 3.1 MockServerのソースコード転送
**Windows側のターミナル (ローカル環境)** を開き、SCPコマンドを使ってローカルのMockServerソースコードをサーバーへ送付します。
*(※以下はMockServerのローカルソースが `n:\antigravity\jepx4\mock_server` にある場合の仮定です。実際のパスに合わせてください)*

```powershell
# 1. SSHでサーバー側にフォルダを作成する指令を送ります
ssh -i "jepx-stage-key.pem" azureuser@<パブリックIP> "mkdir -p ~/mock_server"

# 2. scpコマンドで mock_server配下のファイルをサーバーへ転送します
scp -i "jepx-stage-key.pem" -r n:\antigravity\jepx4\mock_server\* azureuser@<パブリックIP>:~/mock_server/
```

### 3.2 Dockerイメージのローカルビルドとコンテナ起動
ふたたび **サーバー側のターミナル (SSH)** に戻ります。転送されたソースコード内には `Dockerfile` が含まれている前提とします。

```bash
cd ~/mock_server

# 1. 転送したソースからDockerイメージをビルド (名前を jepx-mock-image:latest とする)
docker build -t jepx-mock-image:latest .

# 2. ビルドしたイメージを使ってコンテナをバックグラウンド(-d)で起動する
# ホストマシンの 8888 (Web API 等) と 9999 (Stomp 等) ポートにバインドして起動します。
# (※コンテナ側のポートは実際のDockerfile内のEXPOSE指定に合わせて調整してください)
docker run -d --name jepx-mockserver -p 8888:8888 -p 9999:9999 jepx-mock-image:latest
```

### 3.3 コンテナの動作確認
コンテナ起動後、`curl` コマンドでMockServerが内部で生きているか確認します。

```bash
# 起動中コンテナの確認
docker ps

# Mock APIの応答確認 (パスは実際のものに合わせてください)
curl http://127.0.0.1:8888/
```

## 🚀 4. JEPX アプリケーションのデプロイ (ファイル転送と設定)

### 4.1 プロジェクトファイルの転送 (SCPによる手動デプロイ)
**Windows側のターミナル (ローカル環境)** を新しく別に開き、SCPコマンドを使ってローカルのソースコードを丸ごとサーバーへ送付（手動デプロイ）します。

```powershell
# 1. まずSSHでサーバー側にフォルダを作成する指令を送ります
ssh -i "jepx-stage-key.pem" azureuser@<パブリックIP> "mkdir -p ~/jepx_project"

# 2. scpコマンドで jepx_project配下のファイルをサーバーへ転送します
# (※ .venv などのローカル専用の特大フォルダは予め除外しておくか、必要な中身だけを選択して送ってください)
scp -i "jepx-stage-key.pem" -r n:\antigravity\jepx4\jepx_project\* azureuser@<パブリックIP>:~/jepx_project/
```

### 4.2 サーバー側でのパッケージインストール
ふたたび **サーバー側のターミナル (SSH)** に戻り、転送されたコードの起動セットアップを行います。

```bash
# プロジェクトフォルダへ移動
cd ~/jepx_project

# 仮想環境（閉鎖されたPythonの小部屋）を作成し、中に入る
python3 -m venv .venv
source .venv/bin/activate

# 必要なライブラリ (Django, PyYAML, authlib, uvicorn 等) をサーバーにもインストール
pip install -r requirements.txt
```

### 4.3 セキュア設定 (.settings) の構成
ソースコード内に書けないSSOの秘密情報をセットします。

```bash
mkdir -p .settings
nano .settings/sso_config.yml
```

表示された真っ黒なエディタ（nano）に、Step 1.3 で取得した情報を記載します。
```yaml
ENTRA_TENANT_ID: "メモしたDirectory (tenant) IDの値"
ENTRA_CLIENT_ID: "メモしたApplication (client) IDの値"
ENTRA_CLIENT_SECRET: "作成したクライアントシークレット(Secret)の値"
```
*(入力が終わったら、キーボードの `Ctrl + O` を押し、`Enter` で保存します。その後 `Ctrl + X` で閉じます)*

### 4.4 静的ファイルの収集 (マイグレーション不要)
今回構築するシステムはDBレス環境のため `python manage.py migrate` は不要です。Nginxが読み出しやすいように静的ファイルのみを集約します。

```bash
# 検証環境用(stage)の設定であることをシステムに教える
export DJANGO_SETTINGS_MODULE=config.settings.stage
export SECRET_KEY="long-random-secret-key-for-verification"

# 静的ファイル(CSS/JS/画像等)を一箇所に集める (-noinputは確認をスキップするため)
python manage.py collectstatic --noinput

# 念のため設定エラーがないかチェック
python manage.py check
```

## ⚙️ 5. Uvicornの常駐化（Systemdサービス設定）

ユーザーがいつアクセスしてきても応答できるように、SSH画面を閉じてもアプリが24時間サーバーの裏側で動き続けるように（デーモン化）設定します。

```bash
sudo nano /etc/systemd/system/jepx-stage.service
```

以下の内容をペーストして保存します（`Ctrl + O` → `Enter` → `Ctrl + X`）：
```ini
[Unit]
Description=Uvicorn daemon for JEPX Web UI (Stage DB-less)
After=network.target

[Service]
User=azureuser
Group=www-data
WorkingDirectory=/home/azureuser/jepx_project
Environment="DJANGO_SETTINGS_MODULE=config.settings.stage"
Environment="SECRET_KEY=your-secure-secret-key"

# 仮想マシンのローカルポート8000番で、ASGIサーバーであるUvicornを起動
ExecStart=/home/azureuser/jepx_project/.venv/bin/uvicorn config.asgi:application --host 127.0.0.1 --port 8000

# プログラムがクラッシュしても自動で再起動
Restart=always

[Install]
WantedBy=multi-user.target
```

サービスをシステムに登録し、起動させます。
```bash
sudo systemctl daemon-reload
sudo systemctl start jepx-stage
sudo systemctl enable jepx-stage

# 正しく Active: active (running) になっているか確認します (qを押して終了)
sudo systemctl status jepx-stage
```

## 🚪 6. Nginx ＆ HTTPS(Let's Encrypt) の設定

Django(8000番)へ外部からのアクセスをルーティングし、通信を `https://` (ポート443) で安全に暗号化します。

### 6.1 Nginx リバースプロキシ設定
Nginxの専用設定ファイルを作成します。
```bash
sudo nano /etc/nginx/sites-available/jepx_project
```

以下の内容をペーストして保存します：
```nginx
server {
    listen 80;
    server_name jepx-stage.mkuwan.jp; # Step1.2でDNSに設定したドメインを指定

    # 静的ファイルの配信 (collectstaticされた置き場)
    location /static/ {
        alias /home/azureuser/jepx_project/staticfiles/;
    }

    # 上記以外はすべて内部のポート8000(Uvicorn/Django)へ転送
    location / {
        proxy_pass http://127.0.0.1:8000;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }
}
```

設定を有効にしてNginxを再起動します。
```bash
# デフォルトで入っている不要な設定を消す
sudo rm -f /etc/nginx/sites-enabled/default

# 作成した設定のショートカットファイルを配置して有効化
sudo ln -s /etc/nginx/sites-available/jepx_project /etc/nginx/sites-enabled/

# Nginxの文法チェックが successful であることを確認
sudo nginx -t

# Nginxを再起動
sudo systemctl restart nginx
```

### 6.2 SSL証明書の取得とHTTPS化 (Certbot)
無料のSSL証明書配布サービス（Let's Encrypt）から証明書を自動取得し、先程のNginx設定をHTTPS用に自動で書き換えてもらいます。

```bash
# Nginx用のCertbotプラグインを実行
sudo certbot --nginx -d jepx-stage.mkuwan.jp
```
*(※初回のみ、メールアドレスの入力と規約同意(A)を聞かれます。完了すると「Congratulations!」と表示され、自動でNginxが再起動＆HTTPS化されます)*

## 🎉 7. 稼働検証・テスト

すべての構築が完了しました！お手元のWebブラウザから以下の手順で動作確認を行ってください。

1. **アクセス確認**: アドレスバーに `https://jepx-stage.mkuwan.jp/` と入力し、JEPXのWeb画面が安全な通信（鍵マーク）で表示されるか確認します。
2. **SSO 連携確認**:
   - 画面右上の **「Entra ID ログイン」** ボタンをクリックします。
   - 見慣れた Microsoft のログイン画面に切り替わるので、組織のアカウント情報を入力して認証を進めます。
3. **認証後のセッション動作確認**:
   - ログイン後、本システムのトップ画面へ自動的に戻り、画面右上に**自分のユーザー名**が表示されることを確認します。
   - この状態（ログイン済み）でのみ、DAH/ITDなどの各種市場照会・入札ページへアクセスできることを確認します（MockServerと通信し、正常にデータが描画されるはずです）。
4. **ログアウトリダイレクトの確認**:
   - 右上の「ログアウト」ボタンを押します。
   - ブラウザのCookieが消去され未認証状態に戻り、各種保護ページへアクセスしようとすると自動的に再度 `/login/` の画面へ飛ばされる（弾かれる）ことを確認したら、検証は100点満点で完了です！
