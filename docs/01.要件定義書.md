# 00. 要件定義（JEPX API連携システム）

## 1. 文書情報

### 1.1 文書目的
本書は、JEPX（日本卸電力取引所）の会員サイトにおけるWeb画面運用廃止、およびAPI連携のみの移行に伴い、翌日市場取引および時間前市場取引をシステム化・自動化するための要件を定義する。

### 1.2 参照文書
- `00.指示書.md`
- `901.JEPX 専用接続線 接続技術書.md`（ver.2.1）
- `902.API仕様書(翌日市場取引システム).md`（ver.1.1.1）
- `903.API仕様書(時間前市場取引システム).md`（ver.1.0.3）

### 1.3 用語
- DAH: 翌日市場（Day-Ahead）
- ITD: 時間前市場（Intraday）
- ITN: 市場情報通知（配信）
- 約定: 成立した取引
- Graph API: Microsoft 365（SharePoint等）と連携するためのAPI
- JP1: 統合システム運用管理ツール（バッチ処理のジョブスケジューラ）
- ステートレス: 処理結果・照会データをローカルに永続保持せず、必要都度 JEPX API から取得して表示・処理する設計方針
- 監査ログ: 操作者・実行時刻・API 入出力・判定理由等を記録するログ（DB ではなくログファイル）

### 1.4 基本方針（アーキテクチャ原則）

本システムは以下の 3 原則を全要件の前提とする。これと矛盾する要件を設けてはならない。

| 原則 | 内容 |
|------|------|
| **ステートレス（DB不使用）** | 業務データ表示や処理に必要なデータは、都度 JEPX API から取得（またはSharePointから取得）する。リレーショナル DB等の永続データストアは使用しない。コード定義は設定ファイル（YAML/JSON）で管理する。 |
| **画面開発なし（UIはExcel）** | Webブラウザ向けのGUI画面は作成しない。ユーザーインターフェースはすべてExcel（VBA等）およびSharePointを介して提供される。 |
| **証跡のログ保存** | 操作・API 通信・判定・エラーの証跡を監査ログとして保存する。システムの証跡はすべてログファイルに求め、DB 代わりの利用はしない。 |

## 2. 背景・目的

### 2.1 背景
- 現行運用は Web 画面操作と RPA を中心に、入札・削除・照会・約定参照・結果比較を実施している。
- JEPX は GUI ベース運用を終了し API 中心に移行するため、現行 RPA （画面スクレイピング等）は継続不能となる。
- 現行業務は、証跡保管・再送手順・通信障害対応が個別運用に依存しており、システム化による標準化が急務である。

### 2.2 目的
- Excel / CSV の計画値ファイルから API 実行、結果照会、結果ファイルのSharePoint出力までを一貫して自動化する。
- 翌日市場はJP1からの日次バッチ実行により人手を介さず確実に処理を完了させる。
- 時間前市場は、ユーザーが使い慣れたExcel（VBA）をWebクライアントとして活用し、素早い入札・照会・板情報の取得を実現する。
- 冪等性・再送制御・監査証跡・障害復旧（BCP）の仕組みを標準化し、実運用可能な品質を確保する。
- **DB を使用せず、処理はすべてステートレスとすることで、運用・保守コストを排除する。**

### 2.3 成功指標（KPI）
- 定常日の入札処理成功率: 100%（翌日市場では午前9時の入札完了を絶対要件とする）
- API 送信重複・二重入札事故: 0 件
- 監査ログ欠落: 0 件

## 4. To-Be 業務像

### 4.1 全体業務フロー（翌日市場 バッチ）
1. ユーザーが入札計画値ファイル（CSV/Excel）をSharePointに配置する。
2. JP1のスケジューラによりバッチコマンド（Python）が起動し、Graph APIで計画値を取得する。
3. 形式・業務ルール・スケジュール妥当性を検証する。
4. API リクエスト電文を生成し、JEPX API（DAH1001等）への送信・応答受信を行い、ログに記録する。
5. JEPX APIから入札照会／約定照会／清算照会データを取得する。
6. 比較・集計・レポートファイルを生成し、Graph APIを利用してSharePointの所定フォルダへ出力する。
7. 【例外時】エラー内容をSharePoint上にエラーレポートとして出力し、監視チームへ通知（ログ監視経由）する。

### 4.2 全体業務フロー（時間前市場 Excel API）
1. ユーザーが手元のExcel（VBA）上で入札情報や照会条件を入力し、実行する。
2. Excel VBAがWebサーバー（Django API）に対してHTTPリクエストを送信する。
3. Djangoアプリはリクエストを検証し、JEPX ITD/ITN APIを呼び出して送受信を行う。
4. JEPXからの応答や板情報を、Excelが解釈可能な形式（JSON）で即座に返却する。
5. Excel側でデータをパースし、シート上に結果・状態を描画する。
6. 【例外時】JEPX APIから受けたエラーやシステム側のエラーは、Django APIのレスポンスとしてExcelへ返却し、画面上にダイアログ等で提示する。

### 4.3 例外時フロー・リカバリ
- API エラー発生時: エラー詳細（入力不備/通信異常など）をログへ出力。
- 再送可否判定: いかなる場合も、JEPX 照会 API（DAH1002/ITD1003等）を呼び出して既送信の有無を確認し、未到達（未登録）の場合のみ再送を行う。
- 復旧: プロセスの一時停止・再起動後も、SharePointのファイルや該当日の照会データから業務の継続が可能なステートレス性を担保する。

## 5. システム構成要件

### 5.1 技術要件（指示書準拠）
- アプリケーション: Python（Django および ASGI構成のUvicorn等）
  - ※ リアルタイム配信（ITN）をメモリストリームで中継するため、ASGI構成を必須とする。
- バッチスケジューラ: JP1 (または同等のジョブキック基盤) + Pythonスクリプト
- 外部連携: Microsoft Graph API (SharePoint共有用)
- インフラ: AWS 環境（VPC 構成、クライアント用意）
- 接続経路: バッチ＆Web API連携サーバー → FW → 中継サーバー → AWS Direct Connect → JEPX
- データストア: DB 不使用。ログはファイル出力する。

### 5.2 論理コンポーネント
- SharePoint連携モジュール（Graph API I/O）
- Excel API通信モジュール（HTTPリクエスト/レスポンス、ITNストリームの中継）
- 取込・検証モジュール（メモリ内処理）
- JEPX API 通信モジュール（Socket/TLS/gzip/独自電文生成・解析）
- 業務制御モジュール（締切管理、再試行、冪等性・二重送信チェック）
- 監査ログモジュール（操作・API 入出力・判定の記録）
- 出力ファイル生成モジュール（比較結果CSV/Excel、PDF）

### 5.3 データの永続化方針
| 種別 | 扱い |
|------|------|
| 計画値（CSV/Excel入力） | バッチ: SharePointから取込後のみメモリ内処理。 / API: HTTPリクエスト内で処理。 |
| API 送受信データ | すべて監査ログに出力する（マスキング済） |
| 照会結果・約定結果 | 都度 JEPX API から取得して、SharePointファイルやExcel連携にそのまま使用。 |
| レポート・出力ファイル | API取得データを用いて一時生成し、SharePointへ出力（またはExcelへ返却）後に破棄。 |
| コード定義・設定値 | YAML/JSON/ENV ファイルでソースコードと共に管理 |
| ログ | AWS CloudWatch等 またはローカルファイルに出力する |

## 6. 外部インターフェース要件

### 6.1 JEPX接続方式（専用接続線）
- 通信方式: TCP/IP Socket 通信
- 暗号化: TLS 1.3
- IP 制限: 各サーバ毎に許可 IP は 1 つ
- Socket 上限: 1 通信線・1 サーバ当たり `5 + 1（配信用は1）` に準拠
- 一般通信: 3 分無通信でサーバ側切断
- 配信通信: 無通信でもサーバ側切断なし

### 6.2 JEPX電文フォーマット
- 構成: `SOH` + ヘッダ + `STX` + ボディ + `ETX`
- ヘッダ文字コード: ASCII（`項目名=値` をカンマ区切り）
- 要求ヘッダ項目: MEMBER（会員ID）, API（機能番号）, SIZE（ボディ Byte 数）
- 応答ヘッダ項目: STATUS（電文ステータス）, SIZE（ボディ Byte 数）
- ボディ: gzip 圧縮された JSON

### 6.3 電文ステータス対応
- `00`: 電文処理正常（機能エラーはボディ status を参照）
- `10`: 電文フォーマット異常
- `11`: 会員ID権限なし
- `19`: サーバシステム異常（JEPX側の異常）

### 6.4 共通 API（接続維持）
- SYS1001（ソケット接続時間延長）を実装し、一般通信で 3 分アイドルになる前に keep-alive 送信を行うこと。

### 6.5 Microsoft Graph API 連携
- 認証方式: OAuth 2.0 クライアント・クレデンシャル・フロー（ユーザー操作不要）
- 利用エンドポイント: SharePoint サイトに対する Files / Drives API（Read/Write）。

## 7. 業務要件（BR）

### BR-01 対象業務完全性
翌日市場・時間前市場の入札、削除、照会、約定、清算の業務を API で実行し、結果をファイル（SharePoint）やExcelを通じて完全に代替できること。

### BR-02 自動化（DAH）
翌日市場の入札作業は JP1 等から起動するバッチ処理によって完全自動化し、締切時間（9時）までに安全確実に入札を完了させること。

### BR-03 ユーザビリティ（ITD）
時間前市場業務はExcelをUIとして一通り実施でき、状況確認・入札・削除・照会がスムーズに行えること。

### BR-04 監査性
すべての受信（SharePointやExcelからの）、JEPXに対する送信、応答、判定、時刻をログに出力すること。

### BR-05 事業継続性
通信障害・処理失敗時に、SharePointのファイルと JEPX 照会 API を用いた現況確認を起点として、バッチやAPIリクエストの再実行による復旧が可能であること。

### BR-06 ステートレス運用
状態を抱え込まないステートレス設計とし、バッチの進行ステータス等を除きRDB等へ保存しないこと。

## 8. 機能要件（FR）

### 8.1 取込・検証

#### FR-01 ファイル取込・受付
- DAH: Graph APIを用いてSharePointから計画値（CSV/Excel）を取得できること。
- ITD: Excel VBAからのHTTPリクエストパラメータ（JSON等）を受け付けること。

#### FR-02 入力バリデーション
- JEPXが定める必須・形式・範囲・単位・コード・日程・不整合を検証すること。
- JEPX エラー分類（required / format / inconsistency / unit / range / code / schedule 等）と一致した検証ロジックを実装し、不備がある場合は事前に弾き出すこと。

#### FR-03 業務妥当性
- 入札価格が 10 の倍数であることを検証すること。
- 入札量の小数処理（小数第1位有効、2位以下切捨）を統一フォーマット化すること。

### 8.2 API 実行（翌日市場）

#### FR-10 DAH API 実装
以下 API を実行可能とする。
- DAH1001 入札
- DAH1002 入札照会
- DAH1003 入札削除
- DAH1004 約定照会
- DAH9001 清算照会

#### FR-11 DAH 入札制御
- 取込データに基づく通常入札の投入順序、再実行順序を制御すること。
- 各 API の送受信内容をすべて監査ログに記録すること。

### 8.3 API 実行（時間前市場）

#### FR-20 ITD/ITN API 実装
以下 API を実行可能とする。
- ITD1001 入札
- ITD1002 入札削除要求
- ITD1003 入札照会
- ITD1004 約定照会
- ITD9001 清算照会
- ITN1001 市場情報通知受信

#### FR-21 ITD 特有制御
- 入札削除要求（DEL）の扱いを通常入札と区別してロジック制御すること。

#### FR-22 配信受信（ITN中継）
- 市場情報通知（CONTRACT / BID-BOARD）をDjango (ASGI) アプリケーションで継続受信すること。
- 受信データをメモリ内の共有領域（キュー等）に一時保持し、ExcelクライアントからのポーリングやWebSocket/SSEの要求に応じて滞りなく返却できること。永続保存（Redis等）はしないこと。
- 0件通知（情報なし）を正常ケースとして扱うこと。

### 8.4 照会・表示・出力制御

#### FR-30 都度照会
- 入札照会・約定照会・市場結果照会は、要求（バッチの次のステップ、またはExcelからのリクエスト）のたびに JEPX API を呼び出して最新データを取得すること。

#### FR-31 比較・検算レポート
- 取り込んだ計画値（メモリ上）と JEPX API から取得した約定値を照合し、数量・価格・金額の差分を計算した結果ファイルを生成（SharePoint出力 / Excel返却）すること。

#### FR-32 清算・通知書等出力
- 清算照会（DAH9001/ITD9001）を都度発行し、応答に含まれる PDF（Base64形式等）や清算データをExcel/PDFファイルとして所定位置へ出力すること。

### 8.5 実行制御・冪等性

#### FR-40 二重送信防止
- バッチ処理やExcel API双方において、同一対象への重複送信を防ぐため、実行前に JEPX 照会 API（DAH1002/ITD1003等）を呼び出してJEPX側の既登録状態を確認し、未登録（未到達）の場合のみ再送（新規入札）を行うこと。
- この二重送信確認・照会操作の経緯をすべてログに出力すること。

#### FR-41 再試行
- 通信失敗時（TLSエラー等）の自動再試行は、設定回数を上限にプロセス内で実施すること。
- 手動再処理（バッチ再キック等）の場合もFR-40の既済チェックを必須とする。

## 9. 非機能要件（NFR）

### 9.1 性能
- 翌日市場対象のバッチ処理は、エラー時の自動再試行分を含め、規定された午前9時までに確実に入札処理が完了するスループットを有すること。
- 時間前市場のExcel向けAPIは、長時間のブロッキングを避け、JEPXからの応答遅延時には適切にタイムアウトし、クライアントに返答すること。

### 9.2 可用性
- 通信障害時やJEPX側システムのエラー時に、内部でハンドリングしクラッシュしないこと。
- 再起動時のリカバリにおいて、照会 API を状態確認の起点とし、過去のプロセス状態（DB情報）に依存せず処理再開が可能であること。

### 9.3 信頼性
- 電文フレーム（SOH/STX/ETX）、ヘッダ SIZE 判定、gzip の破損検証等のプロトコル異常への堅牢性を持つこと。
- STATUS ヘッダ及びボディの status フィールド双方を用いて成否判定を厳密に行うこと。

### 9.4 セキュリティ
- JEPX連携には TLS1.3 を用いること（実行OS・Pythonディストリビューション側で確実に対応させること）。
- 機密情報（SharePointのClient Secret等のAPIキー類）は、ソースコードに直書きせず、環境変数やAWS Secrets Manager等で管理すること。
- Excel APIは社内網を想定するため専用認証はないものの、ファイアウォール（IP制限）等でアクセスコントロールを行うこと。

### 9.5 ログ
- すべての証跡はログ（ファイル等）で管理し、再送結果や不正アクセスを追跡可能であること。
- APIリクエスト時の機密な認証情報(ID/パスワード等)はマスクして出力すること。

### 9.6 保守性
- JEPX API の仕様変更（エリア増減・商品追加など）に追従できるよう、コード内にハードコーディングせず、設定ファイル（YAML/JSON）等で管理すること。

## 10. ログ要件

本システムは DB を使用しないため、証跡・監査・障害調査はすべてログファイルに依存する。

### 10.1 ログ種別

| 種別 | 記録内容 | 用途 |
|------|----------|------|
| 業務・操作ログ | JP1バッチ起動/終了、SharePoint連携結果、Excel APIリクエスト受信、対象データキー | 監査・証跡 |
| API 通信ログ | JEPX API 種別、リクエスト内容（マスキング済）、レスポンス STATUS、ボディ status、処理時間 | 障害調査・監査 |
| エラーログ | エラー分類（format, code等）、詳細メッセージ、スタックトレース | 障害調査 |
| 実行ログ（バッチ）| 処理対象日、処理ファイル名、成功/失敗件数 | 運用監視 |

### 10.2 ログ出力・設定方針
- Pythonの `logging` モジュール等により、対象サーバー内に一次保管する（ローテーション設定必須）。
- 保管期間など長期アーカイブについては、運用設計またはAWS CloudWatch Agent等への転送で担保する。

### 10.3 禁止事項
- 平文での認証情報出力。
- アプリケーションデータストア（DB）代わりのログ利用の一切の禁止。

## 11. コード定義・設定管理

### 11.1 設定ファイル管理対象
- エリアコード（1〜9）、時間帯コード（01〜48等）
- エリアグループコード（A1, A2, A4, A7, A9等 ※時間前市場向け）
- 入札種別コード（SELL-LIMIT, BUY-LIMIT, SELL-MARKET, BUY-MARKET, FIT, 承認電源, DEL等）
- 取引中断コード（0, 1）や通知種別コード（CONTRACT, BID-BOARD）
- API エンドポイント、タイムアウト秒、再試行回数

### 11.2 管理方針
- アプリケーション起動時にキャッシュし、設定変更時はファイルの更新およびプロセスの再デプロイ/再起動により反映する。

## 12. バッチ要件（翌日市場）

### 12.1 実行シナリオ
1. JP1からバッチコマンド起動（日付・対象ファイル等を引数とする）
2. SharePointより計画値ファイル取込（メモリ内）
3. 検証（メモリ内）
4. 重複送信（再送）チェックのための照会（JEPX API）
5. 入札送信（JEPX API）
6. 約定照会・清算照会（JEPX API）
7. メモリ上の計画値と約定値の比較・レポートファイル生成
8. レポートファイルをSharePointに出力
9. 操作・通信記録をログに出力して完了通知（JP1へ戻り値返却）

### 12.2 制御要件
- 単一のプロセスで全て完了させるか、障害切り分けのためフェーズ毎にキックするかはジョブ設計に委ねるが、排他制御（同一対象日時の重複実行防止）を行うこと。
- リカバリ処理は全て「JEPX照会APIでの現況確認の上での未処理分再送」方式に統一する。

## 13. API要件（時間前市場 Excel向け）

### 13.1 インターフェース構成（内部API）
- Djangoサーバーは時間前市場の「司令塔・中継器」として動作する。
- RESTful相当のエンドポイント（`/api/v1/itd/bid`, `/api/v1/itd/inquiry` 等）を設計すること。

### 13.2 リアルタイム取得（ITN連携）
- 配信通信による板情報等の受信状況をExcelが随時取得できるよう、最新の情報を滞りなく返却する設計とする。

### 13.3 レスポンスハンドリング
- 通信エラー、必須項目未入力、JEPX側の業務エラー等、全てのエラー要因をExcel（VBA）で判別可能な内部ステータス・メッセージに変換のうえJSONで返却し、画面ダイアログ表示に繋げること。

## 15. エラー・例外要件

### 15.1 エラー分類と対応
- **入力エラー**（`required` / `format` / `unit` / `range` / `code`）
  - 検証で弾き、SharePointにエラーログファイルとしてフィードバック、またはExcelAPIレスポンスとして返す。再送は原因修正後に実行。
- **業務整合・期間・会員エラー**（`inconsistency` / `schedule` / `member`）
  - 入札不可能な時間外やデータの論理矛盾、会員権限の不備。同様に修正を促す。
- **電文・通信エラー**（STATUS `10` / `11` / `19`、Socket/TLS異常）
  - ネットワーク遅延やJEPXの一時高負荷。内部で規定回数リトライし、それでも失敗した場合は運用側にエスカレーション。

### 15.2 二重対策と隔離
- エラー起因でバッチが落ちても、再起動時に正常なデータ（既にJEPXで受領されたもの等）に再送を掛けない冪等性制御を行う。
- 一部の行データのエラーによってファイル全体が止まるか、正常な行だけ先通しするかは仕様策定で決定する（通常はスキップ等して警告・隔離する）。

## 16. 受入要件（抜粋）

### 16.1 機能受入
- 翌日市場の取込〜入札〜照会〜比較出力までの一連の自動化がJP1バッチ起点で成立すること。

- ステートレス設計として、プロセス再起動や別端末からの再操作において一貫した最新データが参照できること。

### 16.3 運用受入
- 通常時の運用手順書・SharePointファイルの配置手順が成立すること。
- 障害発生時の切り分け・復旧（再実行）手順がスムーズに行えること。

## 18. 要件定義まとめ

本要件定義では、JEPX API 移行において、GUIを用いない（ExcelVBA + SharePoint + JP1バッチ構成）という条件下での業務自動化・システム化要件を定義した。

本システム最大の特徴は**「ステートレス・DB不使用」**であり、業務データは常に JEPX API（またはSharePoint）から取得することで構成の運用・保守コストを最小化できる。

中核要件：
- **Excel/SharePoint をフロントエンドとした自動化の成立**（DAH/ITD）
- **JEPX 専用接続仕様の完全準拠**（TLS1.3・Socket制約等）
- **DBなし・ログによる完全な証跡担保**
- **冪等性・JEPX事前照会による二重送信の防止**

以降の基本・詳細設計フェーズでは、本書に基づき各モジュールの責任分解およびインターフェース定義を実施する。
