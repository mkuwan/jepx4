### 2.2 環境別Settings設計（prod / stage / dev）

Django Settingsを3ファイルに分離し、環境変数 `DJANGO_SETTINGS_MODULE` で切り替える。

#### 2.2.1 環境定義

| 環境 | Settings | JEPX接続先 | MEMBER | 用途 |
|------|---------|-----------|--------|------|
| **dev** | `config.settings.dev` | `127.0.0.1:8888` (MockServer) | `9999` | ローカル開発・単体テスト |
| **stage** | `config.settings.stage` | JEPX検証環境IP:Port | `9999` | JEPX検証環境での結合テスト |
| **prod** | `config.settings.prod` | JEPX本番IP:Port | `0841` (実会員ID) | 本番運用 |

#### 2.2.2 base.py（共通設定）

``
**[共通設定のロジック仕様]**
- プロジェクトの基準ディレクトリパスを定義する。
- 統合アプリケーション群（通信クライアント、バッチ、API、ストリーム、SharePoint連携）を依存モジュールとして登録する。
- JEPXとの通信要件に従い、通信タイムアウト(30秒)、Keep-Alive間隔(150秒)、一般リクエストのコネクション上限(5本)とITN配信用上限(1本)、及び通信エラー時のリトライ上限(3回)とその待機基底時間を定義する。
- SharePoint連携に必要な認証情報（Tenant ID, Client ID/Secret, Site ID等）を環境変数経由でロードする。
- 監査ログおよびシステムログ群（api_comm, audit, error）のローテーション出力先やフォーマット、機密情報マスキングフィルターの設定を紐付ける。
- バリデーションにおいて、1件でもエラーが発生した際に即座に処理を取りやめる「フェイルファスト」フラグを定義する。\n
``

#### 2.2.3 dev.py（開発環境 — MockServer接続）

``
**[開発環境 (dev) のロジック仕様]**
- デバッグモードを有効化し、ホスト検証を全て許可する。
- 接続先をローカルのMockServer（127.0.0.1:8888）に向ける。
- TLS証明書の厳密な検証を無効化するか、自己署名証明書ファイル（MockServer用）を指定して通信を行う。
- SharePoint連携はスキップし、入出力ファイルはプロジェクト内のローカルテストディレクトリ（test_data）を参照する。
- ログ出力をファイルだけでなく標準出力(コンソール)にも流し、開発時の視認性を高める。\n
``

#### 2.2.4 stage.py（検証環境 — JEPX検証環境接続）

``
**[検証環境 (stage) のロジック仕様]**
- デバッグモードを無効化する。
- 接続先をJEPX提供の検証サーバー（IPアドレスおよびポート）へ向ける。
- TLS検証フラグを有効にし、JEPXから提供された検証用ルート証明書を読み込んで安全な通信を担保する。
- SharePoint連携を有効化する。\n
``

#### 2.2.5 prod.py（本番環境 — JEPX本番接続）

``
**[本番環境 (prod) のロジック仕様]**
- デバッグモードを完全に無効化し、許可された正規ホスト名（ALLOWED_HOSTS）以外のアクセスを遮断する。
- 接続先をJEPX本番環境とし、機密情報である実際のログインメンバーID（0841等）やIP情報を環境変数から読み込む。
- TLS検証を有効にし、OS側の証明書ストア等を利用した厳格な証明書検証を実施する。
- リトライ上限回数を開発環境より余裕をもって引き上げる（5回）。
- パフォーマンスやセキュリティの観点からコンソール出力を停止し、ファイルへのロギングに限定する。\n
``

#### 2.2.6 環境切り替え方法

``
**[コマンド: 環境別起動手順]**
- 環境変数 `DJANGO_SETTINGS_MODULE` に各環境のモジュール（dev, stage, prod）を指定する。
- Django内蔵サーバー（開発用）や ASGI サーバー（Uvicorn・本番用）にホスト名やポートを指定してプロセスを起動する。検証・本番環境では必ず環境変数を外から注入する仕様とする。\n
``

## 第3章 JEPX Socket通信基盤 詳細設計

### 3.1 モジュール構成

**[構造図論理: モジュール構成]**
アプリケーションの主幹通信群は主に5つの責務に分割されたオブジェクトで協調動作する。
- 外部APIから汎用的に呼ばれる「JepxApiClient（送信・制御の総合窓口）」
- 通信経路の枯渇を防ぐ「ConnectionPool（セッション再利用プーリング機構）」
- 低レベルなTCP層を担う「JepxConnection（TLSソケット制御）」
- ペイロードをJEPX規格に翻訳する「JepxProtocol（電文圧縮・フォーマット解析）」
- 無通信タイムアウトを防止する「KeepAliveManager（自動死活監視デーモン）」

### 3.2 protocol.py — 電文組立・解析

``
**[電文組立・解析のロジック仕様]**
- **組立処理 (build_packet)**
  1. 業務要求内容をJSON文字列に変換し、UTF-8バイト列とする。
  2. gzip圧縮アルゴリズムを用いて圧縮し、サイズ(SIZEヘッダ用)を算出する。
  3. 会員ID(MEMBER)、API機能番号(API)、算出したサイズ(SIZE)をカンマ区切りで結合し、ヘッダ部を生成する。
  4. SOH(開始文字) + ヘッダ部 + STX(テキスト開始) + gzip圧縮データ + ETX(終了文字) の順番でバイナリを結合し、最終的な通信パケットとして返却する。
  
- **解析処理 (parse_response)**
  1. 受信したバイトデータから SOH, STX, ETX の位置を検索し、パケットが完全な形で到着しているかを判定する（欠落時は通信例外）。
  2. SOHとSTXに挟まれたヘッダ部を文字列解析し、キーと値のペアにマッピングする。
  3. STXとETXに挟まれたボディ部の実バイト数と、ヘッダに記載された SIZE の値が一致するかを検証する。
  4. ボディ部をgzipとして解凍し、そのJSON文字列をプログラム内で扱える辞書オブジェクトに変換して返却する。
  
- **ステータス検証 (validate_status)**
  1. レスポンスヘッダに含まれる STATUS の内容を精査する。
  2. '00' (正常)以外であった場合、それぞれの要因（10: フォーマット異常、11: 権限不足、19: システム異常）に対応する専用の例外を発生させて後続処理へ通知する。\n
``

### 3.3 exceptions.py — 例外クラス体系

``
**[例外クラス体系のロジック仕様]**
- 通信およびAPIの各段階で発生するエラーを体系的に定義する。
- **JepxFormatError / JepxAuthError**: ヘッダ仕様に違反した場合のエラー。これらは送信データの見直しが必要であるため「自動リトライ不可（retryable = False）」とする。
- **JepxSystemError / JepxConnectionError / JepxTimeoutError**: JEPXシステムの過負荷、Socket切断、応答のタイムアウトなど、一時的なインフラ要因によるエラー。時間を置けば成功する可能性が高いため「自動リトライ対象（retryable = True）」とする。
- **JepxBusinessError**: 通信自体は成功したが、JSONボディ内に含まれるステータスが "200" 以外（例: バリデーション違反や重複エラー）であった場合のエラー。これも「自動リトライ不可」とする。\n
``

### 3.4 client.py — JepxApiClient 送受信

``
**[JEPX通信クライアントのロジック仕様]**
- **初期化**: 環境設定に基づき、相手先ホスト、同時接続上限、TLS証明書設定を組み込んだ「コネクションプール」を用意し、リトライ回数の上限等を保持する。
- **リクエスト送信 (send_request)**
  1. 業務データをプロトコル組立機能に渡し、送信用パケットを準備する。
  2. 再試行ループを開始する。
  3. コネクションプールから空き接続（Socket）を取得する。
  4. パケットを送信し、JEPXからの返信を待機・受信する。タイムアウトを計測する。
  5. プロトコル解析機能を呼び出し、ステータスやデータサイズに異常がないか確認する。
  6. エラーが発生せず、正常に処理が完了した場合はプールへ接続を返却し、ボディ部のデータをシステムへ返却する。
  7. もし例外が発生した場合、その例外が「リトライ可能」であれば数秒（指数バックオフで時間延長）待機後に同じ手順を最初から再実行する。「リトライ不可」あるいは上限回数に達した場合は処理を強制終了して異常を通知する。\n
``

``

### 3.7 コネクションプール設計

**[構造図論理: コネクション状態遷移]**
コネクションのライフサイクルは4つのフェーズを遷移する仕様。
- 初期状態（Idle）から通信処理が割り当てられると使用中（InUse）に。
- 通信が完了して解放されるとIdleへ戻る。
- Idleのまま一定時間（150秒など）が経過した場合、監視状態（KeepAlive）に切り替わり、JEPXへ死活確認（SYS1001）を送信する。
- 監視で応答がない場合、または異常発生時は即座に破棄（Closed）されプールから除外される。

- **一般通信 (DAH/ITD)**: プール上限 5 本。`acquire()` で取得し、使用後に `release()` で返却。
- **配信通信 (ITN)**: 専用 1 本。プールとは別管理。Keep-Alive不要（配信通信は無通信切断なし）。
- **Keep-Aliveデーモン**: プール内のIdle接続に対し、最終通信から150秒経過でSYS1001を自動発行。

### 3.8 keepalive.py — Keep-Aliveデーモン

``
**[TLSソケット接続（JepxConnection）のロジック仕様]**
- **SSLContextの制御**: JEPX設定に応じて、OSの証明書ストアを利用するか、指定した証明書ファイルを利用するか、証明書自体の検証をスキップして暗号化通信だけを行うかを動的に切り替える。
- **Socket接続・送信**: 設定されたホスト・ポートにTLS接続を行い通信チャネル(reader/writer)を確立する。
- **パケット受信**: 非同期処理でデータを一定サイズごとに読み出し、バッファにつなぎ合わせる。JEPX電文の終端を示す ETX制御文字 (0x03) を検知した瞬間に読み取りを完了として結果を返す。
- **接続監視**: 切断された場合やタイムアウトした場合には適切にリソースを開放する。\n
``

### 4.2 排他制御（同一受渡日の重複実行防止）

01.要件定義書 §12.2 に基づき、同一受渡日のバッチ重複実行を防止する。

``
**[コネクションプールのロジック仕様]**
- 限られた接続数を使い回す（マルチプレキシング）ためのプール。
- **取得 (acquire)**: 既存の接続リスト内に空いている接続（通信待機可能）であればそれを再利用する。空きがなく、接続上限に達していなければ新規に接続を張ってリストに追加する。上限に達している場合はどれかが空くまで待機ロックを掛ける。
- **返却 (release)**: 通信処理が終わった接続を使用中リストから解放し、次回の再利用可能リストへと戻す。
- **一括破棄**: 万一の通信全断やアプリケーション終了時にはすべての保持しているSocketの終端処理を呼び出し、リストをクリアする。\n
``

**使用例 (`dah_bid.py` 内):**
``
**[Keep-Alive デーモンのロジック仕様]**
- JEPXシステムは、一定時間（標準3分）通信がないセッションを強制的に切断する仕様を持つ。
- このデーモンは非同期のバックグラウンドタスクとして常駐し、指定された間隔（例: 150秒）ごとに起動する。
- プール内に保持されている全てのコネクションを走査し、最終使用時刻からの経過時間をチェックする。
- しきい値を超えている「アイドル状態」のコネクションがあれば、それらに対して死活監視用のAPI通信（SYS1001）を強制的に送信する。
- 応答が正常であれば接続を維持し、失敗した場合はそのコネクションを無効として切断・プールから除外する。\n
``

> ※ JP1側でも排他ジョブグループの設定で二重防止することを推奨するが、アプリ側でもファイルロックを設けることで手動再実行時の事故を防止する。

### 4.3 dah_bid コマンド 処理フロー

**[構造図論理: dah_bid コマンド処理フロー]**
翌日市場バッチの業務フロー論理：
1. キック要因（JP1等）から指定された日付をもとにGraph API（SharePoint連携）経由で計画値情報を取得。
2. CSVデータを内部辞書へパース後、バリデーションで値の完全性を確認（エラー時はフェイルファストで処理中止しレポート出力）。
3. JEPX側へ現在日（対象日）の照会要求を流し、すでに入力データ群が「登録済（送信済）」かをチェックする冪等性担保フローを通す。
4. 入力未済であればJEPXへ入札電文を送信する。
5. 成功すれば結果をログや後続バッチへ流し、異常なら指数バックオフによる自動再試行またはアラートを上げる。

### 5.3.1 ITD入札の二重送信防止（FR-40準拠）

01.要件定義書 FR-40 は「Excel API双方において」二重送信防止を求めている。ITD側は以下の2層で防止する。

**第1層: Excel VBA側（UIレベル）**
- 入札ボタン押下後、即座にボタンを `Enabled = False` に設定し、API応答またはタイムアウトまで再押下を不可にする。

**第2層: Django View側（ビジネスロジックレベル）**
``
**[入札バッチ (dah_bid) のロジック仕様]**
- CUIの引数として対象日（`--date`）を受け取り動作する。
- フェイルファストな実行を前提とし、以下のステップを踏む：
  1. 排他ロックを取得する。（既に同じ日の処理が動いていれば中断）
  2. SharePoint連携機能（ローカルテスト時はファイルシステム）から、指定日の入力元となる入札用CSVファイルをダウンロードする。
  3. ファイル内容をパースし、各行に対して共通・DAH専用のバリデーションチェックを適用する（小数丸めや上限チェック等の仕様V-001~016）。
  4. 1件でもエラーがあれば即座にエラーファイル（CSV）と監査ログ・アラートを出力し、入札を実行せずに全処理を中断する（FAIL_FAST仕様）。
  5. エラーがなければ、通信クライアントを介して JEPX (DAH1001) へバルクで入札要求コマンドを送信する。
  6. すべて正常に受理されれば、結果の受付番号等を反映し、処理完了とする。
  7. 最後にファイルロックを解放する。\n
``

> ※ DAHバッチ（§4.6）と同じ「事前照会による既登録確認」パターンを採用している。

#### InMemoryStore インターフェース

``
**[ITD API 入札ビュー・照会ビューのロジック仕様]**
- **ITD 入札 (POST /api/v1/itd/bid)**: Excel VBA等のフロントエンドからJSONフォーマットの入札条件を受信し、バリデーションを経由したのち非同期通信でJEPX (ITD1001) へ注文を発行。冪等性チェック（同一データの二重入札抑止）を含み、受付番号付きの結果JSONをシリアライザを通してクライアントへ返す。

#### ITN受信バックグラウンドタスク

``
**[ITN 市場情報 (Stream) 配信のロジック仕様]**
- このAPIはロングポーリングまたはServer-Sent Events (SSE) 技術を用いてブラウザやExcel側へリアルタイムの市場情報を中継する機能。
- **InMemoryStore との連携**: バックグラウンドで常駐するITNタスクが JEPXから受信し続けている全情報（板情報・約定履歴）はDjangoサーバーの「メモリ上（Store）」にバージョン番号付きでストックされている。
- **バージョン・差分配信方式**: クライアントからのリクエスト（GET /api/v1/itn/stream）に付与されたバージョン番号と比較する。
  - もしクライアントがもつバージョンがサーバーの最新版に等しければ、情報が更新されるまで非同期で一定時間（あるいは更新があるまで）待機する。
  - もし遅延していれば、現在サーバーが保持している状態との差分、または蓄積された最新の全量データと新しいバージョン番号を即座にJSONで返却する。\n
``

### 5.5 ASGI起動時のITNタスク起動

``
**[ITN 受信デーモン (ITN Receiver) のロジック仕様]**
- ASGIアプリケーション起動と同時にバックグラウンドで独立して稼働する無限ループのプロセス。
- 通信クライアントを用いて専用のAPI識別子 (ITN1001) でJEPXサーバーへ接続を確立し、市場状況や取引約定データを**プッシュ形式**で受信し続ける。
- 受信した更新データを順次解釈し、データストア（InMemoryStore）の「板情報（BID-BOARD）」または「約定情報（CONTRACT）」へマージ・追記を行う。
- 更新が完了するたびにStoreのバージョン番号をインクリメントし、通知待機中のクライアントAPI側にイベントを発火させて新規データが存在することを知らせる。
- 接続が断絶した場合は通信エラーを検知し、数秒の待機の後に自動でチャネルを作り直し、初回は全量データの再取得（スナップショット）からのリスタート処理を実行する。\n
``

### 5.6 Excel VBA クライアント側実装設計

ITD/ITN Web APIを利用するExcelマクロ側の実装方針を定めます。

#### 5.6.1 共通要件とライブラリ
*   **通信コンポーネント**: `MSXML2.XMLHTTP60` または `WinHttp.WinHttpRequest.5.1` を使用して同期・非同期のHTTP通信を行います。
*   **JSONパース/ビルド**: 外部ライブラリである `VBA-JSON` (https://github.com/VBA-tools/VBA-JSON) を標準モジュールとしてインポートし、JSON文字列とDictionary/Collectionオブジェクトの相互変換を行います。
*   **APIベースURL設定**: 環境（本番、検証、開発）ごとのAPIベースURLをExcelの非表示シートや設定シート（例:`Config`シート）に持ち、VBAコード内では動的に参照させます。

#### 5.6.2 ITD入札API 呼び出し実装例 (VBA)

UI（入札ボタン）から呼び出されるITD入札の処理フローです。二重送信防止（FR-40）のため、送信中はボタンを無効化します。

``
**[VBA: Excel側からのITD入札呼び出しロジック例]**
- `MSXML2.XMLHTTP60`（いわゆるXMLHttpRequest）オブジェクトを用いて、Djangoが提供する非同期WebAPI (`POST /api/v1/itd/bid`) へHTTPアクセスを行う。
- リクエスト生成：入札日時、入札方向（売り/買い）、エリアコードなどをJSONフォーマットのテキストとして構築する。
- リクエスト送信：ContentTypeを"application/json"に設定し、同期的に送信する。
- 受信と描画：Djangoからのレスポンス結果を受け取り、ステータスが200であれば入札完了、それ以外の異常コードであればVBA側でエラーダイアログを表示する等の制御を行う。一定時間（30秒など）応答が途絶えた場合のサーバーダウン対処もVBA側の例外ハンドリングに持たせる。\n
``

#### 5.6.3 ITN市場情報通知 ポーリング実装例 (VBA)

Excel VBAではSSE (Server-Sent Events) を非同期で維持することが難しいため、一定間隔でのポーリング (`mode=poll`) を利用します。
Windows API の `SetTimer` を利用するか、`Application.OnTime` を利用してバックグラウンドで更新を掛けます。

``
**[SharePoint 認証 (OAuth2) のロジック仕様]**
- Microsoft 365 の Graph API を用いた連携機能の前段階となる認証情報の取得。
- クライアントID（環境変数）やテナントIDを用いて、Microsoftの認可サーバーエンドポイント (`login.microsoftonline.com/{テナントID}/oauth2/v2.0/token`) へ認証要求をPOSTする。
- クライアントクレデンシャル方式による認証を採用し (`grant_type=client_credentials`)、自身に対するトークンを取得する。
- 取得したアクセストークンはメモリ上に一定時間（失効する約1時間）キャッシュされ、後続のファイル入出力を伴うGraph APIの全てのリクエストの Authorization ヘッダに `Bearer` トークンとして付与される設計とする。\n
``

#### 5.6.4 Excel VBA側のエラーハンドリング方針

APIからのレスポンスHTTPステータスおよび `error_code` に基づき、VBA側で以下のユーザーメッセージを表示・制御します。

| HTTPステータス | レスポンス内 error_code | VBAでのユーザー向けメッセージ例 | アクション |
|---------------|-------------------------|--------------------------------|------------|
| `200 OK` | (なし/success=true) | 処理が完了しました | 成功処理・UI更新 |
| `400 Bad Request` | `VALIDATION_ERROR` | 入力項目を確認してください。詳細: {message} | UIで入力見直しを促す |
| `400 Bad Request` | `JEPX_BUSINESS_ERROR` | JEPX側の業務処理エラーです。詳細: {message} | 入札条件の見直し |
| `409 Conflict` | `DUPLICATE_BID` | 同一条件の入札が既に送信済みです。(bidNo={...}) | UIで変更・削除を促す |
| `502/503/504` | `JEPX_*_ERROR` | JEPX接続で一時的なシステムエラーが発生しました。 | 少し待ってからの再試行 |
| タイムアウト等例外 | (レスポンス取得失敗) | サーバーとの通信に失敗しました。 | ネットワーク確認 |

### 7.3 小数切捨ての挙動（V-013）

JEPX仕様により「小数第2位以下は切り捨て」とされているが、本システムではバリデーション時に**自動補正**を行い、補正した事実を**監査ログに記録**する。エラーとはしない。

|-------|-------|--------|
| `100.56` | `100.5` | `[OPERATION] V-013 volume自動補正: row=3, 100.56→100.5` |
| `100.5` | `100.5` | (出力なし) |

この方針は、JEPX側でも切捨て処理が行われるための整合性確保と、運用担当者への透明性提供を両立する。
