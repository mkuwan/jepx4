## 1. 前提条件・用語整理

### 1.1 各環境の定義

| 環境名 | 英語名 | 接続先 JEPX | 使用設定ファイル | 用途 |
|--------|--------|------------|----------------|------|
| 開発環境 | development (dev) | MockServer（ローカル起動）| `config.settings.dev` | 機能開発・ユニットテスト |
| 検証環境 | staging (stage) | JEPX 検証サーバー（本物）| `config.settings.stage` | 結合テスト・受入テスト |
| 本番環境 | production (prod) | JEPX 本番サーバー（本物）| `config.settings.prod` | 実業務運用 |

### 1.2 開発環境の前提ソフトウェア

開発環境（ローカルPC）では以下のソフトウェアが必要です。作業前に必ずインストールされていることを確認してください。

| ソフトウェア | 必要バージョン | 確認コマンド | 入手先 |
|-------------|--------------|------------|-------|
| Python | 3.11 以上 | `python --version` | https://www.python.org/downloads/ |
| Docker Desktop | 最新安定版 | `docker --version` | https://www.docker.com/products/docker-desktop/ |
| Git | 最新版 | `git --version` | https://git-scm.com/ |

> Docker Desktop を起動していないと、後述の MockServer が起動できません。**PC 起動後に必ず Docker Desktop を立ち上げてください。**

### 1.3 リポジトリのクローン（初回のみ）

まだリポジトリを取得していない場合は、次のコマンドで取得します。すでに取得済みの場合はこの手順はスキップしてください。

```powershell
# 任意のフォルダへ移動（例: n:\antigravity）
cd n:\antigravity

# リポジトリをクローン
git clone https://github.com/mkuwan/jepx4.git

# クローンされたフォルダへ移動
cd jepx4
```

## 2. 開発環境の実行手順

開発環境では、JEPX本体の代わりに **MockServer** をローカルPC上で起動し、アプリケーションが通信相手として使用します。

```
[ローカルPC]
  ├─ MockServer (Docker)  ←→  ├─ Django アプリケーション
  └─ ブラウザ             →→      (Uvicorn/manage.py runserver)
```

### 2.1 初回環境構築（2回目以降はスキップ）

初めてこのプロジェクトを動かす場合のみ実施します。

#### 2.1.1 MockServer 用の TLS 自己署名証明書の生成

MockServer は TLS（HTTPS相当の暗号化）接続を使用します。ローカルで動かすための自己署名証明書を生成します。

**ターミナル（Windows PowerShell）を開き、以下の手順を実施します。**

```powershell
# プロジェクトのルートフォルダへ移動
cd n:\antigravity\jepx4

# MockServer フォルダへ移動
cd MockServer

# certs フォルダを作成
mkdir -p certs

# OpenSSL を使って自己署名証明書を生成します（有効期限: 3650日 = 約10年）
# ※ openssl コマンドが使えない場合は Git Bash または WSL から実行してください
openssl req -x509 -newkey rsa:2048 -keyout certs\server.key -out certs\server.crt -days 3650 -nodes -subj "/CN=jepx-mockserver"
```

上記コマンドが完了すると、`MockServer\certs\` フォルダ内に以下の 2 つのファイルが生成されます：
- `server.crt` ─ 証明書ファイル
- `server.key` ─ 秘密鍵ファイル

> **注意**: `server.key` は秘密鍵ですので、Git などでリポジトリに含めないよう注意してください（`.gitignore` に記載されているはずです）。

#### 2.1.2 Python 仮想環境の作成とライブラリのインストール

```powershell
# jepx_project フォルダへ移動
cd n:\antigravity\jepx4\jepx_project

# Python 仮想環境（.venv）を作成します
python -m venv .venv

# 仮想環境を有効化します（PowerShell の場合）
.\.venv\Scripts\Activate.ps1

# コマンドプロンプト（cmd）の場合は次のコマンドを使用してください
# .\.venv\Scripts\activate.bat

# プロンプトの先頭が (.venv) に変わったらアクティベート成功です
# 必要なライブラリを一括インストールします
pip install -r requirements.txt
```

インストールが完了したら、バージョンを確認して正常にインストールされているか確認します。

```powershell
# Django のバージョン確認
python -c "import django; print(django.__version__)"
# → 5.1.4 などと表示されれば OK
```

#### 2.1.3 SSO 設定ファイルの作成

開発環境では SSO（Microsoft Entra ID）との連携は必須ではありませんが、設定ファイルは必要です。

```powershell
# jepx_project フォルダ内に .settings フォルダを作成
mkdir n:\antigravity\jepx4\jepx_project\.settings

# テキストエディタで sso_config.yml を作成します（VS Code の場合）
code n:\antigravity\jepx4\jepx_project\.settings\sso_config.yml
```

作成したファイルに以下の内容を貼り付けて保存します。開発環境では仮の値で構いません。

```yaml
ENTRA_TENANT_ID: "dev-tenant-id"
ENTRA_CLIENT_ID: "dev-client-id"
ENTRA_CLIENT_SECRET: "dev-client-secret"
```

#### 2.1.4 ログ保存フォルダの作成

```powershell
# logs フォルダを作成します（存在する場合は何も起こりません）
mkdir n:\antigravity\jepx4\jepx_project\logs
```

### 2.2 MockServer の起動

JEPX API をシミュレートする MockServer を Docker で起動します。

> **前提**: Docker Desktop がすでに起動していることを確認してください。タスクバーの通知領域に Docker のクジラアイコンが表示されていれば起動中です。

#### 2.2.1 MockServer フォルダへ移動

```powershell
cd n:\antigravity\jepx4\MockServer
```

#### 2.2.2 Docker Compose を使って MockServer を起動

```powershell
# --build オプション付きで起動するとソースコードの変更が反映されます
# 初回は必ずこのコマンドを使ってください（イメージのビルドから実行します）
docker compose up --build -d
```

このコマンドは以下のことを自動で行います：
1. `Dockerfile` を読み込んで Docker イメージを構築する
2. 構築したイメージからコンテナを作成する
3. `-d`（デタッチ）オプションにより、ターミナルを占有せずにバックグラウンドで起動する

#### 2.2.3 MockServer の起動確認

コンテナが正常に起動したか確認します。

```powershell
# 起動中のコンテナ一覧を確認します
docker ps

# 出力例（STATUS が "Up" になっていれば正常です）:
# CONTAINER ID   IMAGE                    COMMAND           STATUS        PORTS
# abc123def456   mockserver-jepx-mockserver   "python -u main.py"   Up 5 seconds   0.0.0.0:8888->8888/tcp
```

> `STATUS` 欄が `Up X seconds` または `Up X minutes` となっていれば、MockServer は正常に起動しています。

MockServer が起動しているかをポートへの接続で確認します（省略可）。

```powershell
# PowerShell での接続テスト（ポート 8888 が開いているか確認）
Test-NetConnection -ComputerName localhost -Port 8888

# TcpTestSucceeded: True と表示されれば OK
```

#### 2.2.4 MockServer のログ確認

起動に問題がある場合、ログを確認します。

```powershell
# コンテナのログをリアルタイムで確認します（Ctrl+C で停止）
docker compose logs -f

# 直近50行のみ確認したい場合
docker compose logs --tail=50
```

### 2.3 MockServer CA 証明書のエクスポート（初回のみ）

MockServer の TLS 証明書を、Django アプリが信頼するための証明書ファイルとして `jepx_project/certs/` へコピーします。

```powershell
# MockServer フォルダへ移動していることを確認（2.2の手順から続けている場合は既にここにいます）
cd n:\antigravity\jepx4\MockServer

# 証明書エクスポートスクリプトを実行します
python export_cert.py
```

正常に完了すると次のようなメッセージが表示されます：

```
[OK] MockServer CA証明書をエクスポートしました
  FROM: n:\antigravity\jepx4\MockServer\certs\server.crt
  TO:   n:\antigravity\jepx4\jepx_project\certs\mockserver_ca.pem

dev.py の USE_MOCKSERVER_CERT = True で証明書検証付きテストが可能です。
```

`jepx_project\certs\mockserver_ca.pem` が作成されたことを確認します。

```powershell
# ファイルの存在を確認します
Test-Path n:\antigravity\jepx4\jepx_project\certs\mockserver_ca.pem
# True と表示されれば成功です
```

### 2.4 Django アプリケーションの起動

#### 2.4.1 仮想環境の有効化（ターミナルを新たに開いた場合）

ターミナルを開き直した場合や、仮想環境がまだ有効になっていない場合は有効化します。

```powershell
cd n:\antigravity\jepx4\jepx_project

# 仮想環境を有効化します（プロンプトの先頭に (.venv) が付きます）
.\.venv\Scripts\Activate.ps1
```

#### 2.4.2 起動前の設定確認

```powershell
# 開発環境用の設定を使用することを宣言します
$env:DJANGO_SETTINGS_MODULE = "config.settings.dev"

# 設定に問題がないかチェックします
python manage.py check

# 問題がなければ "System check identified no issues." と表示されます
```

> `System check identified no issues.` 以外のメッセージが表示された場合は、エラー内容を確認・対処してから次の手順へ進んでください。

#### 2.4.3 Uvicorn を使ったアプリケーション起動

JEPX アプリケーションは ASGI 対応のサーバー（Uvicorn）で動作します。**Django 標準の `runserver` コマンドではなく、必ず Uvicorn を使って起動してください**（ITN配信ストリームが ASGI 機能を使用しているためです）。

```powershell
# DJANGO_SETTINGS_MODULE が設定されていることを再確認
echo $env:DJANGO_SETTINGS_MODULE
# config.settings.dev と表示されれば OK

# Uvicorn でアプリを起動します
# --reload オプションにより、ソースコードを変更すると自動で再起動します（開発時に便利）
uvicorn config.asgi:application --host 127.0.0.1 --port 8000 --reload
```

起動が成功した場合、次のようなログが表示されます：

```
INFO:     Started server process [XXXX]
INFO:     Waiting for application startup.
INFO:     Application startup complete.
INFO:     Uvicorn running on http://127.0.0.1:8000 (Press CTRL+C to quit)
```

> このターミナルはアプリが動いている間、占有されます。**ターミナルを別ウィンドウでもう1枚開いて**、ブラウザでのアクセス確認など他の操作を行ってください。

### 2.5 動作確認

#### 2.5.1 ブラウザでのアクセス確認

Uvicorn が起動したら、お使いのブラウザ（Chrome や Edge など）を開き、アドレスバーに次の URL を入力してアクセスします。

```
http://127.0.0.1:8000/
```

JEPX のトップ画面（ログイン画面またはダッシュボード）が表示されれば、アプリケーションは正常に起動しています。

#### 2.5.2 MockServer との通信確認

開発環境では DAH（翌日市場）や ITD（時間前市場）の各画面から疑似的なデータを取得できます。

1. ブラウザで `http://127.0.0.1:8000/` にアクセスします。
2. 各市場の画面（例: DAH 入札確認画面）にアクセスします。
3. MockServer が自動的に疑似データを返し、画面にデータが表示されることを確認します。

MockServer が起動していない状態でアクセスすると「接続エラー」が発生します。その場合は 2.2 の手順に戻って MockServer を起動してください。

#### 2.5.3 Uvicorn のログ確認

アプリ起動中のターミナルに、アクセスのたびにリクエストログが表示されます。エラーが出ていないかを確認します。

```
INFO:     127.0.0.1:XXXXX - "GET / HTTP/1.1" 200 OK   ← 正常
INFO:     127.0.0.1:XXXXX - "GET /static/... HTTP/1.1" 200 OK
```

HTTP ステータスコードが `200` や `302` であれば正常です。`500` や `400` が表示された場合はエラーが発生しているため、詳細なログを確認してください。

### 2.6 停止手順

#### 2.6.1 Django アプリケーションの停止

Uvicorn を起動しているターミナルで、次のキー操作を行います。

```
Ctrl + C
```

`Shutdown complete.` または `Finished server process` などのメッセージが表示されれば、アプリケーションは安全に停止しました。

#### 2.6.2 MockServer の停止

```powershell
cd n:\antigravity\jepx4\MockServer

# コンテナを停止します（データは保持されます）
docker compose stop

# ── または ──

# コンテナを停止して削除します（次回起動時に再作成されます）
docker compose down
```

> `docker compose stop` はコンテナを止めるだけで再起動時に素早く起動できます。
> `docker compose down` はコンテナを削除するためより完全な停止ですが、次回は再作成に少し時間がかかります。

## 3. 検証環境の実行手順

検証環境は Azure 仮想マシン（VM）上で稼働しています。システムは `systemd` サービスとして登録されており、VM の起動と同時に自動的に起動するよう設定されています。

> **検証環境の接続先**: 本物の JEPX 検証サーバーに接続します。MockServer は使用しません。

```
[検証環境 Azure VM]
  ├─ Nginx（80/443番ポートで外部リクエストを受付）
  │    └→ 127.0.0.1:8000 へプロキシ転送
  ├─ Uvicorn + Django（systemd サービス: jepx-stage）
  └─ JEPX 検証サーバー（外部）へ TLS接続
```

### 3.1 Azure VM への SSH 接続

検証環境の VM に接続します。接続に必要な鍵ファイル（`.pem` ファイル）を事前に担当者から入手してください。

**お手元の Windows PC でターミナル（PowerShell）を開き、次のコマンドを実行します。**

```powershell
# 鍵ファイル (.pem) が保存されているフォルダへ移動します
# 例: ダウンロードフォルダに保存した場合
cd ~\Downloads

# SSH で VM に接続します
# <パブリックIPアドレス> の部分は実際の VM の IP アドレスに置き換えてください
ssh -i "jepx-stage-key.pem" azureuser@<パブリックIPアドレス>
```

接続を試みると次のような確認メッセージが表示される場合があります。

```
The authenticity of host '20.xxx.xxx.xxx (20.xxx.xxx.xxx)' can't be established.
Are you sure you want to continue connecting (yes/no/[fingerprint])?
```

**`yes` と入力して Enter キーを押してください。**

接続が成功すると、プロンプトが以下のような表示に変わります。

```
azureuser@jepx-stage-vm:~$
```

この表示になればサーバー内での操作が可能です。

### 3.2 環境変数の確認・設定

検証環境では、接続先 JEPX サーバーの情報を環境変数で管理しています。

#### 3.2.1 現在の設定値を確認

```bash
# 現在の環境変数を確認します
echo "DJANGO_SETTINGS_MODULE: $DJANGO_SETTINGS_MODULE"
echo "JEPX_HOST: $JEPX_HOST"
echo "JEPX_PORT: $JEPX_PORT"
echo "SECRET_KEY 設定有無: $([ -n "$SECRET_KEY" ] && echo '設定済み' || echo '未設定')"
```

#### 3.2.2 systemd サービスに設定されている環境変数の確認

環境変数はシステムサービス起動設定（systemd Unit ファイル）に記載されています。内容を確認します。

```bash
# systemd サービスの設定ファイルを確認します
sudo cat /etc/systemd/system/jepx-stage.service
```

表示された内容の `[Service]` セクション内の `Environment=` 行を確認します。以下のような変数が設定されているはずです。

```ini
Environment="DJANGO_SETTINGS_MODULE=config.settings.stage"
Environment="SECRET_KEY=your-secure-secret-key"
Environment="JEPX_HOST=xxx.xxx.xxx.xxx"
Environment="JEPX_PORT=8888"
Environment="ALLOWED_HOST=jepx-stage.mkuwan.jp"
```

#### 3.2.3 環境変数の変更が必要な場合

接続先 IP アドレスやポートを変更する必要がある場合は、次の手順で設定ファイルを編集します。

```bash
# サービス設定ファイルをテキストエディタで開きます
sudo nano /etc/systemd/system/jepx-stage.service
```

- キーボードの矢印キーで変更したい行へ移動します。
- 値を修正します（例: `JEPX_HOST` の値を新しい IP アドレスに変更）。
- 編集完了後、`Ctrl + O` を押して保存します。`Enter` で確定します。
- `Ctrl + X` を押してエディタを終了します。

**設定ファイルを変更した場合は、必ず次のコマンドで変更を反映させてサービスを再起動してください。**

```bash
# 設定変更を systemd に読み込ませます
sudo systemctl daemon-reload

# サービスを再起動します
sudo systemctl restart jepx-stage
```

### 3.3 各サービスの起動確認と起動

#### 3.3.1 全サービスのステータス確認

まず、各サービスが現在どのような状態にあるかを確認します。

```bash
# Django アプリ（Uvicorn）のステータスを確認します
sudo systemctl status jepx-stage

# Nginx のステータスを確認します
sudo systemctl status nginx
```

どちらも `Active: active (running)` という緑色の表示になっていれば正常稼働中です。

**表示例（正常時）:**
```
● jepx-stage.service - Uvicorn daemon for JEPX Web UI (Stage DB-less)
     Loaded: loaded (/etc/systemd/system/jepx-stage.service; enabled; ...)
     Active: active (running) since ...
   Main PID: XXXX (uvicorn)
      Tasks: X (limit: XXXX)
     Memory: XX.XM
```

`q` キーを押すと確認画面を閉じることができます。

#### 3.3.2 サービスが停止していた場合の起動手順

`Active: inactive (dead)` や `Active: failed` と表示されている場合は、次の手順でサービスを起動します。

```bash
# Django アプリ（Uvicorn）を起動します
sudo systemctl start jepx-stage

# 起動後すぐにステータスを再確認します
sudo systemctl status jepx-stage
```

```bash
# Nginx を起動します
sudo systemctl start nginx

# 起動後すぐにステータスを再確認します
sudo systemctl status nginx
```

#### 3.3.3 自動起動設定の確認

VM の再起動後に自動でサービスが起動するように設定されているかを確認します。

## 4. 本番環境の実行手順

> ⚠️ **警告**: 本番環境のシステムへの変更は、実際の電力取引業務に直接影響します。操作前に必ず担当責任者の承認を得てから実施してください。計画外の再起動や停止は厳禁です。

本番環境の基本的な手順は検証環境と同じですが、接続する JEPX サーバーが本番系に変わり、使用する設定ファイルが `config.settings.prod` になります。

```
[本番環境 Azure VM]
  ├─ Nginx（80/443番ポートで外部リクエストを受付）
  │    └→ 127.0.0.1:8000 へプロキシ転送
  ├─ Uvicorn + Django（systemd サービス: jepx-prod）
  └─ JEPX 本番サーバー（外部）へ TLS接続
```

### 4.1 Azure VM への SSH 接続

本番環境の VM に接続します。接続に必要な鍵ファイル（`.pem` ファイル）を事前に担当者から入手してください。

```powershell
# 鍵ファイルが保存されているフォルダへ移動します
cd ~\Downloads

# SSH で本番 VM に接続します
# <本番VM-IPアドレス> の部分は実際の本番 VM の IP アドレスに置き換えてください
ssh -i "jepx-prod-key.pem" azureuser@<本番VM-IPアドレス>
```

接続確認のメッセージが表示されたら `yes` と入力して Enter を押します。

```
azureuser@jepx-prod-vm:~$
```

このプロンプトが表示されれば、本番 VM への接続成功です。

### 4.2 環境変数の確認・設定

本番環境では、以下の環境変数が必須です。すべて設定されていることを確認してください。


`[Service]` セクションに次の変数がすべて適切な値で設定されていることを確認します。

| 変数名 | 説明 | 設定例 |
|--------|------|--------|
| `DJANGO_SETTINGS_MODULE` | 使用する設定ファイル（本番用）| `config.settings.prod` |

| `JEPX_HOST` | JEPX 本番サーバーのIPアドレス | `xxx.xxx.xxx.xxx` |

| `JEPX_MEMBER_ID` | JEPX 会員ID（実契約の会員ID）| `0841` 等 |

> **重要**: `SECRET_KEY` は本番環境と検証環境で異なる値を使用してください。本番の `SECRET_KEY` が漏洩した場合、セッション Cookie が偽造される危険があります。

### 4.3 TLS 証明書の配置・確認

本番環境では JEPX 本番サーバーへの TLS 接続に証明書が必要です。

#### 4.3.1 TLS 証明書の配置方法 2 通り

**方式 A（推奨）: OS の信頼済み証明書ストアにインストールする方法**

JEPX から提供されたルート CA 証明書（`jepx_root_ca.pem` など）を OS の証明書ストアに登録します。

```bash
# JEPX ルート CA 証明書を OS の信頼済みストアへコピーします
# ※ 証明書ファイルは別途 JEPX またはネットワーク担当者から入手してください
sudo cp /home/azureuser/jepx_root_ca.pem /usr/local/share/ca-certificates/jepx_root_ca.crt

# 証明書ストアを更新します
sudo update-ca-certificates

# 出力例:
# Updating certificates in /etc/ssl/certs...
# 1 added, 0 removed; done.
```

方式 A の場合、`jepx-prod.service` の `Environment` に `JEPX_TLS_CA_CERT` は設定しません（`prod.py` の設計により OS ストアが自動で使用されます）。

**方式 B（代替）: 証明書ファイルをパスで指定する方法**

```bash
# 証明書ファイルを certs フォルダへコピーします
cp /home/azureuser/jepx_root_ca.pem /home/azureuser/jepx_project/certs/jepx_root_ca.pem

# ファイルのパーミッション（参照権限）を設定します
chmod 644 /home/azureuser/jepx_project/certs/jepx_root_ca.pem
```

`jepx-prod.service` のサービス設定に次の 1 行を追加します。

```ini
Environment="JEPX_TLS_CA_CERT=/home/azureuser/jepx_project/certs/jepx_root_ca.pem"
```

#### 4.3.2 証明書の有効期限確認

```bash
# 証明書の有効期限を確認します（本番運用前に必ず実施）
openssl x509 -in /home/azureuser/jepx_project/certs/jepx_root_ca.pem -noout -dates

# 出力例:
# notBefore=Jan  1 00:00:00 2024 GMT
# notAfter=Dec 31 23:59:59 2026 GMT
# ← "notAfter" が現在日より未来であることを確認してください
```

## 5. 障害発生時の一次対応チェックリスト

問題が発生した際は、以下のチェックリストを上から順番に確認してください。

### 5.1 共通チェック事項

- [ ] サーバー（VM）が起動しているか（Azure ポータルで確認）
- [ ] ネットワーク（インターネット接続）に問題がないか
- [ ] サービスのステータスが `active (running)` になっているか（`sudo systemctl status jepx-xxx`）
- [ ] Nginx のステータスが `active (running)` になっているか
- [ ] エラーログにエラー内容が記録されていないか（`sudo journalctl -u jepx-xxx -n 50`）

### 5.2 よくある問題と対処法

| 症状 | 確認箇所 | 対処法 |
|------|---------|--------|
| 画面が表示されない（502 Bad Gateway）| Uvicorn サービスが動いているか | `sudo systemctl status jepx-xxx` でステータス確認 → 停止していれば起動 |
| 画面が表示されない（接続拒否）| Nginx が動いているか | `sudo systemctl status nginx` で確認 → 停止していれば起動 |
| SSO ログインが失敗する | Entra ID の設定、リダイレクト URI | Azure ポータルでアプリ登録のリダイレクト URI を確認 |
| JEPX 接続エラーが出る | 環境変数 JEPX_HOST / JEPX_PORT の設定 | `sudo cat /etc/systemd/system/jepx-xxx.service` で確認 |
| TLS 証明書エラーが出る | 証明書の有効期限・配置場所 | `openssl x509 -in <証明書ファイル> -noout -dates` で期限確認 |
| 静的ファイル（CSS/画像）が表示されない | `collectstatic` の実行状況 / Nginx の alias 設定 | `python manage.py collectstatic` を再実行 |
| ログが記録されない / ログファイルがない | `logs/` フォルダの存在確認 | `mkdir -p ~/jepx_project/logs` で作成し再起動 |

### 5.3 一次対応でも解決しない場合

以下の情報を収集して、開発担当者または JEPX 接続担当者に連絡してください。

```bash
# 1. サービスの最新ログを収集します（直近200行）
sudo journalctl -u jepx-prod -n 200 > /tmp/jepx_journal_$(date +%Y%m%d_%H%M%S).log

# 2. アプリケーションのエラーログを収集します
cp ~/jepx_project/logs/error.log /tmp/jepx_error_$(date +%Y%m%d_%H%M%S).log

# 3. 収集したファイルをローカルへ転送します（別ターミナル・ローカル側で実行）
scp -i "jepx-prod-key.pem" azureuser@<本番VM-IP>:/tmp/jepx_*.log .
```
